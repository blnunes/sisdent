# Sisdent local pre-production host setup

## Purpose and final result

This guide describes how to prepare a new Ubuntu machine as the dedicated
Sisdent local pre-production host. The intended result is:

- the application runs in Docker and is reachable only from the trusted local
  network;
- Avahi advertises the host as `sisdent-preprod.local`, without requiring a DNS
  record or router configuration;
- Swagger UI is available at
  `http://sisdent-preprod.local/swagger-ui/index.html`;
- `http://sisdent-preprod.local/swagger-ui.html` redirects to the URL above;
- GitHub-hosted runners test the code and build the container image;
- a repository-scoped self-hosted GitHub Actions runner deploys the approved
  image to this machine.

This hostname is an mDNS name for the local network. It is not published to
public DNS or exposed to the internet. mDNS normally works only inside the
same broadcast network or VLAN; routing it between VLANs requires an mDNS
reflector or gateway support.

## Architecture

The host is an execution target, not a development machine. It contains only
Docker, the GitHub Actions runner, runtime configuration, deployment files,
and persistent application data.

The normal deployment flow is:

```text
manually select a branch, tag, or commit in Deploy pre-production
  -> GitHub-hosted runner resolves it to a commit SHA and runs tests
  -> GitHub-hosted runner builds the container image
  -> image is published as ghcr.io/blnunes/sisdent:<commit-sha>
  -> deployment bundle is uploaded as a GitHub Actions artifact
  -> sisdent-preprod self-hosted runner downloads the bundle
  -> Docker Compose starts the approved image
  -> /actuator/health is checked
  -> the deployment is accepted or rolled back
  -> pre-production is tested and approved
  -> reviewed merge to master deploys the approved commit to Render production
```

Java, Maven, Caddy, H2, and the application itself run in containers or on
GitHub-hosted runners. They do not need to be installed directly on the host.
Do not keep a Sisdent source checkout under `/srv/sisdent` or
`/opt/actions-runner`.

## Requirements

### Machine and network

- Ubuntu Desktop or Server 24.04 or newer on x86-64.
- A trusted wired or wireless LAN.
- A stable LAN address. A DHCP reservation is recommended for predictable
  administration, although Avahi itself follows address changes.
- Outbound HTTPS access to `github.com`, `api.github.com`, and `ghcr.io`.
- No inbound port forwarding from the router to ports 22 or 80 on this host.
- An administrator account with `sudo` access.

### GitHub

- Administrator access to the `blnunes/sisdent` repository settings.
- A short-lived runner registration token generated by GitHub during setup.
- Repository secrets required by the existing workflow: `SONAR_TOKEN`,
  `RENDER_API_KEY`, and `RENDER_SERVICE_ID`.
- The workflow in `.github/workflows/ci.yml` on the `master` branch.

The local runner does not need an inbound internet port. It opens outbound
connections to GitHub and uses the workflow's short-lived `GITHUB_TOKEN` to
download artifacts and authenticate to GHCR.

## 1. Inspect and update Ubuntu

Identify the administrator, LAN interface, IPv4 address, and subnet before
changing firewall rules:

```bash
hostnamectl
whoami
ip -4 -brief address
ip -4 route
```

Update the operating system and install the host packages:

```bash
sudo apt-get update
sudo DEBIAN_FRONTEND=noninteractive apt-get upgrade -y
sudo DEBIAN_FRONTEND=noninteractive apt-get install -y \
  ca-certificates curl jq openssh-server ufw unattended-upgrades \
  avahi-daemon libnss-mdns
```

Check whether Ubuntu requires a restart, and restart only during an approved
maintenance window:

```bash
test -f /var/run/reboot-required \
  && echo "REBOOT REQUIRED" \
  || echo "NO REBOOT REQUIRED"
```

## 2. Install Docker Engine

Install Docker Engine from Docker's official Ubuntu repository. Do not install
Docker Desktop and do not configure the Docker daemon to listen on TCP.

```bash
sudo install -m 0755 -d /etc/apt/keyrings
sudo curl -fsSL https://download.docker.com/linux/ubuntu/gpg \
  -o /etc/apt/keyrings/docker.asc
sudo chmod a+r /etc/apt/keyrings/docker.asc

. /etc/os-release
echo "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.asc] https://download.docker.com/linux/ubuntu ${UBUNTU_CODENAME} stable" \
  | sudo tee /etc/apt/sources.list.d/docker.list >/dev/null

sudo apt-get update
sudo DEBIAN_FRONTEND=noninteractive apt-get install -y \
  docker-ce docker-ce-cli containerd.io docker-buildx-plugin \
  docker-compose-plugin

sudo systemctl enable --now docker ssh avahi-daemon
sudo docker run --rm hello-world
docker --version
docker compose version
```

## 3. Configure the local mDNS name

Set the host name and restart Avahi:

```bash
sudo hostnamectl set-hostname sisdent-preprod
sudo systemctl restart avahi-daemon
systemctl is-active avahi-daemon
getent hosts sisdent-preprod.local
```

Avahi advertises `sisdent-preprod.local` directly on the LAN. No router DNS
record, public domain, or internet publication is required. Client devices
must support mDNS and be connected to the same LAN/VLAN. Linux with
`libnss-mdns`, macOS, and devices with Bonjour support normally resolve the
name automatically.

## 4. Create the deployment identity and directories

Create a dedicated user with no password and no sudo access:

```bash
sudo useradd --create-home --shell /bin/bash github-runner
sudo passwd --lock github-runner
sudo usermod --append --groups docker github-runner

sudo install -d -o github-runner -g github-runner -m 0750 \
  /opt/actions-runner /srv/sisdent
sudo install -o github-runner -g github-runner -m 0600 \
  /dev/null /srv/sisdent/runtime.env
```

Membership in the `docker` group is effectively root-equivalent. It is limited
to the dedicated deployment identity and must not be granted to untrusted
users.

Write the following non-secret runtime configuration to
`/srv/sisdent/runtime.env`:

```dotenv
SISDENT_IMAGE_REPOSITORY=ghcr.io/blnunes/sisdent
SISDENT_BIND_ADDRESS=0.0.0.0
```

Then restore ownership and permissions:

```bash
sudo chown github-runner:github-runner /srv/sisdent/runtime.env
sudo chmod 0600 /srv/sisdent/runtime.env
sudo -u github-runner docker ps
```

`0.0.0.0` publishes Caddy on every host interface so another computer on the
LAN can connect. Use `127.0.0.1` instead when access must be restricted to an
SSH tunnel. Binding to the machine's fixed LAN IPv4 address is also valid when
that address is guaranteed not to change.

## 5. Configure SSH and host firewall

Keep SSH on port 22, disable root login, and allow it only from the confirmed
LAN subnet. Replace `192.168.1.0/24` below with the subnet reported by
`ip -4 route` on the new machine.

```bash
sudo ufw default deny incoming
sudo ufw default allow outgoing
sudo ufw allow from 192.168.1.0/24 to any port 22 proto tcp
sudo ufw enable
sudo ufw status verbose
```

Add and test the administrator's SSH public key before disabling password
authentication. Never expose SSH through router port forwarding.

Docker-published ports can bypass ordinary UFW input rules. Before publishing
port 80, restrict Docker forwarding to the trusted LAN. Replace `wlp2s0` and
`192.168.1.0/24` with the confirmed interface and subnet:

```bash
sudo iptables -I DOCKER-USER 1 \
  -m conntrack --ctstate RELATED,ESTABLISHED -j ACCEPT
sudo iptables -I DOCKER-USER 2 \
  -i wlp2s0 -s 192.168.1.0/24 -p tcp \
  -m conntrack --ctorigdstport 80 -j ACCEPT
sudo iptables -I DOCKER-USER 3 \
  -p tcp -m conntrack --ctorigdstport 80 -j DROP
sudo iptables -S DOCKER-USER
```

Persist these rules using the organization's standard firewall mechanism so
they survive a reboot. On a standalone Ubuntu host, one option is:

```bash
sudo DEBIAN_FRONTEND=noninteractive apt-get install -y iptables-persistent
sudo netfilter-persistent save
sudo systemctl enable netfilter-persistent
```

The router must not forward internet traffic to this host. UFW does not need a
general public port 80 allow rule for this design.

## 6. Register the GitHub Actions runner

In GitHub, open:

```text
Repository Settings > Actions > Runners > New self-hosted runner
```

Select Linux and x64. GitHub displays current download and registration
commands. Run those exact commands as `github-runner` inside
`/opt/actions-runner`; do not copy the registration token into documentation,
Git, chat, or a persistent script.

During `config.sh`, use:

- a recognizable runner name, such as `sisdent-preprod`;
- the custom label `sisdent-preprod`;
- the default work directory unless there is a concrete conflict;
- repository scope, not organization-wide scope.

After configuration, install the service from the runner directory:

```bash
cd /opt/actions-runner
sudo ./svc.sh install github-runner
sudo ./svc.sh start
sudo ./svc.sh status
```

Confirm in GitHub that the runner is online and idle and has all required
labels:

```text
self-hosted, linux, x64, sisdent-preprod
```

The runner must not receive sudo privileges. The workflow already grants only
the permissions required for artifact download and GHCR image pull.

## 7. Deployment files and behavior

The GitHub workflow downloads these files to `/srv/sisdent` on every approved
deployment:

```text
compose.preprod.yml
deploy/preprod/Caddyfile
deploy/preprod/deploy.sh
```

Docker Compose creates:

- `data-init`, which prepares volume ownership;
- `app`, which runs the immutable Sisdent image and file-backed H2 database;
- `proxy`, which runs Caddy and publishes host port 80;
- the persistent `sisdent-preprod-data` volume.

Do not remove the named volume during normal cleanup or deployment because it
contains persistent pre-production data.

The deploy script pulls the image tagged with the exact Git commit SHA, starts
the services, and checks `http://127.0.0.1/actuator/health` for up to three
minutes. It records the last healthy image in
`/srv/sisdent/.last-successful-image`. If a later deployment is unhealthy, it
recreates the stack using the last healthy image and leaves the workflow
failed so the event remains visible.

## 8. Trigger the first deployment

Local pre-production deployment is always manual. Pull requests run validation
but do not deploy. A push to `master` deploys only to Render production after
its quality checks pass.

Before pushing, confirm:

```bash
systemctl is-active docker
sudo /opt/actions-runner/svc.sh status
getent hosts sisdent-preprod.local
```

Open `Actions > Deploy pre-production > Run workflow`, enter the branch, tag,
or commit to test, and start the run. GitHub requires this workflow to exist on
the default branch before the button is available. The expected jobs are:

1. `Validate selected revision`;
2. `Build immutable image`;
3. `Deploy selected revision`.

The deployment then validates that `SISDENT_BIND_ADDRESS` is `0.0.0.0` or the
current LAN IPv4 address and requests `/actuator/health` through that LAN
address. The job fails if the service is only bound to loopback or the LAN URL
is not healthy.

The local deployment job waits for a runner with the labels
`self-hosted`, `linux`, `x64`, and `sisdent-preprod`. If the runner is offline,
the job remains queued. A merge to `master` should wait until local
pre-production has been deployed, tested, and approved whenever the change
requires pre-production evidence.

## 9. Validate the deployment

Run these checks on the Ubuntu host:

```bash
docker compose \
  --project-directory /srv/sisdent \
  --env-file /srv/sisdent/runtime.env \
  -f /srv/sisdent/compose.preprod.yml ps

curl --fail http://127.0.0.1/actuator/health
curl --head http://sisdent-preprod.local/swagger-ui.html
curl --fail http://sisdent-preprod.local/swagger-ui/index.html >/dev/null
ss -lnt '( sport = :80 )'
```

The health endpoint should return HTTP 200 with an `UP` status. The short
Swagger URL should return a redirect to `/swagger-ui/index.html`.

From another computer on the same LAN, open:

```text
http://sisdent-preprod.local/swagger-ui/index.html
```

If the name does not resolve on that client, first test the server's LAN IPv4
address directly:

```text
http://<ubuntu-lan-ip>/swagger-ui/index.html
```

If the address works but the name does not, Docker and the application are
healthy and the remaining issue is client-side mDNS support or VLAN isolation.
If neither works, check the Compose services, port 80 listener, Docker firewall
rules, and whether both machines are on the same trusted LAN.

## 10. Operational checks

After setup and after host restarts, verify:

```bash
systemctl is-active docker ssh avahi-daemon
sudo -u github-runner docker ps
sudo ufw status verbose
getent hosts sisdent-preprod.local
curl --fail http://127.0.0.1/actuator/health
curl -sS -o /dev/null -w 'github.com: %{http_code}\n' https://github.com
curl -sS -o /dev/null -w 'api.github.com: %{http_code}\n' https://api.github.com
curl -sS -o /dev/null -w 'ghcr.io: %{http_code}\n' https://ghcr.io/v2/
```

An HTTP 401 from `ghcr.io/v2/` is expected without authentication and confirms
that the registry is reachable.

Review automatic suspend, hibernation, and laptop lid behavior. A dedicated
pre-production host must remain awake while deployments and testing are
expected. Apply power-management changes only after confirming the intended
physical operating mode of the machine.

## Optional one-time manual validation

The current host was initially validated before runner registration by using a
local source checkout to build the `ghcr.io/blnunes/sisdent:preprod` image,
staging the deployment bundle, and starting Compose. This is useful as a
one-time diagnostic procedure, but it is not the normal configuration for a
clean deployment host.

From an existing trusted Sisdent checkout:

```bash
docker build -t ghcr.io/blnunes/sisdent:preprod -f Dockerfile .

sudo install -o github-runner -g github-runner -m 0640 \
  compose.preprod.yml /srv/sisdent/compose.preprod.yml
sudo install -d -o github-runner -g github-runner -m 0750 \
  /srv/sisdent/deploy/preprod
sudo install -o github-runner -g github-runner -m 0640 \
  deploy/preprod/Caddyfile /srv/sisdent/deploy/preprod/Caddyfile
sudo install -o github-runner -g github-runner -m 0750 \
  deploy/preprod/deploy.sh /srv/sisdent/deploy/preprod/deploy.sh

sudo -u github-runner env SISDENT_IMAGE_TAG=preprod docker compose \
  --project-directory /srv/sisdent \
  --env-file /srv/sisdent/runtime.env \
  -f /srv/sisdent/compose.preprod.yml \
  up -d --remove-orphans
```

After the runner is operational, use immutable commit-SHA images supplied by
GitHub Actions and remove the source checkout from the deployment host.

## Security summary

- `sisdent-preprod.local` is local mDNS, not public DNS.
- Do not configure router port forwarding for SSH or HTTP.
- Restrict SSH and Docker-published HTTP traffic to the trusted LAN.
- Do not expose the Docker daemon over TCP.
- Do not store GitHub registration tokens or long-lived GHCR credentials.
- Keep `/srv/sisdent/runtime.env` owned by `github-runner` with mode `0600`.
- Treat the runner and Docker group membership as privileged access.
- Deploy only immutable commit-SHA image tags that passed the Quality Gate.
- Preserve the `sisdent-preprod-data` volume during routine deployments.
